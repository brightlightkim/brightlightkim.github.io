Model-View-Controller(MVC) and Android

All objects in your application must be a model object, a view object, or a controller object.


Model Object

  • Hold the application's data and "business logic."
  • It was designed to model the things in your app is concerned with. ex: user, product in a store, a photo on a server, a television show.
  • No knowledge of the UI
  • Sole Purpose: holding and maintaining data
  • Generally custom classes you create.
  • All of the model objects in your application compose its model layer.

View Object

  • know how to draw themselvews on the screen and how to respond to user input, like touches
  • If you can see it on the screen, then it is a view
  • Android provides various configurable view classes.
  • You can create your custom view classes.
  • An application's view objects make up its view layer.
  • ex: res/layout/activity_main.xml

Controller Objects

  • tie the view and model objects together.
  • contain "application logic."
  • respond to various events triggered by view objects
  • manage the flow of data to and from model objects and the veiw layer.
  • In Android, a controller is typically a subclass of Activity or Fragment.
  • ex: MainActivity

MVC Flow with user input

User Input >> (User interacts with view object) >> Views >> (View sends message to controller) >> Controller >> (Controller updates model Objects) >> Model


Model >> (COntroller takes data from model objects that its views are interested in) >> Controller >> (Controller updates view with changes in model objects) >> Views


Advantages of using MVC

  • Helps with designing and understanding the application as a whole
  • makes classes easier to reuse
